Dataset Explorer

The Dataset Explorer provides a unified interface for two data sources:

🏠 Local Datasets
  • Scans the data/raw/ directory for locally stored EEG datasets
  • Displays subject count, channel configurations, sampling rates, and data size
  • Supported datasets: ADHD-200, TUH EEG, CHB-MIT, Helsinki Neonatal, Sleep EDF
  • Each subject must have a metadata.json file and .edf recording
☁️ EEGDash Remote Streaming
  • Stream 200+ OpenNeuro datasets via the EEGDash API
  • Browse curated catalog organized by disease/area (ADHD, Epilepsy, Sleep, etc.)
  • Auto-configure braindecode models based on dataset characteristics (channels, sampling rate)
  • Metadata and recordings are cached locally for fast re-access
  • Use the Streamed Locally section for previously accessed datasets

Model Discovery

The Braindecode Models page automatically discovers and inspects all models from the installed braindecode.models package.

Features
  • Automatic Discovery — Scans the installed braindecode version for all available model classes (60+ models)
  • Parameter Inspection — View required and optional parameters with default values for each model
  • Code Generation — Generate ready-to-use Python instantiation code snippets with one click
  • Filtering — Filter models by architecture type (Transformer, CNN, Sleep, etc.)
  • Fallback System — When braindecode models have compatibility issues, the system automatically falls back to a built-in SimpleEEGNet

Pipeline Builder

The visual MLOps pipeline builder enables both coders and non-coders to create complete EEG ML pipelines through a drag-and-drop interface.

How It Works
  1. Add Dataset Node — Select a local dataset or stream one via EEGDash
  2. Configure Preprocessing — Add and chain preprocessing steps (filtering, ICA, segmentation, etc.)
  3. Add Split Strategy — Configure train/validation/test split or leave-one-subject-out
  4. Select Model — Choose a braindecode architecture with auto-configured parameters
  5. Configure Validation — Set evaluation metrics and statistical testing options
  6. Connect & Execute — Wire nodes together and run the pipeline with real-time progress
Pipeline Node Reference
Node Type Description Configuration
📂 Dataset Select data source Local path or EEGDash dataset ID
🔽 Filtering Frequency band filtering High-pass, low-pass, notch filter frequencies
📉 Downsampling Reduce sampling rate Target frequency in Hz
⚠️ Bad Channel Detection Detect and interpolate bad channels Threshold and interpolation method
🔗 Re-referencing Change reference scheme Average, REST, bipolar, etc.
🧹 ICA Artifact removal ICA method, eye/muscle component removal
✂️ Segmentation Epoching / windowing Window size, overlap percentage
📏 Baseline Correction Baseline normalization Baseline period in ms (e.g. -200:0)
📊 Data Splitting Train/val/test split Split ratios, stratification, subject-level splitting
🧠 Model Braindecode architecture Model name, hyperparameters, training config
📈 Validation Evaluation metrics Accuracy, F1, AUC-ROC, confusion matrix
Execution & Progress
  • Pipelines run asynchronously in background threads
  • Progress is streamed in real-time via Server-Sent Events (SSE)
  • Run history is persisted to results/pipeline_runs.json
  • Each run stores full metrics, training curves, and model configuration for reproducibility
  • Supports both local datasets and remote EEGDash streams

Evaluation & Results

The Results & Benchmark Hub provides comprehensive model evaluation and comparison.

Available Metrics
  • Classification Metrics — Accuracy, balanced accuracy, F1-score, precision, recall
  • Confusion Matrix — Per-class performance visualization
  • AUC-ROC — Area under the receiver operating characteristic curve
  • Training Curves — Per-epoch loss and accuracy plots
Representation Geometry Analysis
  • Intrinsic Dimensionality — Participation ratio, effective rank, MLE estimate per model per disease
  • CKA Similarity — Centered Kernel Alignment between model representations
  • Manifold Structure — Disease mixing score, cluster purity, k-nearest neighbor overlap
  • Geometry-Performance Correlation — Analysis linking representation geometry to downstream task performance

  • Documentation — Update this documentation or the README
  • Bug Fixes — Check the Issues tab for reported bugs

  • New features include appropriate tests
  • Documentation is updated if needed

  • PR description explains what and why
Project Standards
Code Style
  • Python — Follow PEP 8 guidelines
  • Imports — Group imports: standard library, third-party, local. Use absolute imports.
  • Type Hints — Use Python type hints for all function signatures
  • Docstrings — Use Google-style docstrings for all public modules, classes, and functions
  • Naming — snake_case for functions/variables, PascalCase for classes, UPPER_CASE for constants

Numba JIT Compatibility

To avoid compatibility issues with braindecode models, the platform disables numba JIT compilation at startup:

os.environ['NUMBA_DISABLE_JIT'] = '1'
os.environ['NUMBA_CACHE_DIR'] = '/tmp/numba_cache_disabled'

This is the first thing executed in both flask_app.py and pipeline_executor.py, before any other imports.

Frequently Asked Questions
Place your EEG files (EDF, FIF, etc.) in data/raw/<dataset_id>/ with subject subdirectories. The platform auto-discovers datasets from the data/raw/ folder. If you need a custom loader, add one in src/preprocessing/dataset_loaders.py following the existing pattern.
Numba JIT compilation is disabled globally (NUMBA_DISABLE_JIT=1) to ensure full compatibility with the braindecode model zoo. Some braindecode models use dynamic code paths that conflict with numba's compilation. This is set before any imports in both flask_app.py and pipeline_executor.py.
Activate your virtual environment and run: python src/dashboard/flask_app.py. This starts the Flask development server on http://localhost:5000. Ensure all dependencies are installed via pip install -r requirements.txt first.
The platform uses MNE-Python for data loading, supporting: .edf (European Data Format), .fif (MNE native), .set (EEGLAB), .bdf (Biosemi), .cnt (Neuroscan), .egi (EGI), and more. Files are auto-detected from the data/raw/<dataset_id>/ directory.
The platform automatically detects CUDA-capable GPUs via PyTorch. Ensure you have torch with CUDA support: pip install torch torchvision torchaudio --index-url https://download.pytorch.org/whl/cu118. Check GPU status on the home dashboard System Status panel.
All pipeline results are stored as JSON files in the results/ directory. Each experiment gets its own subdirectory with pretraining_results.json, checkpoints/ for model weights, and geometry_analysis/ for dimensionality/CKA data. Pipeline execution history is stored in results/pipeline_runs.json.
Common causes: (1) Dataset not found — ensure files are in data/raw/<dataset_id>/, (2) Model compatibility — try using SimpleEEGNet as a fallback, (3) Out of memory — reduce batch size or max subjects, (4) Missing dependencies — run pip install -r requirements.txt. Check the run panel for detailed error messages.
No — EEGDash requires an active internet connection to stream datasets from OpenNeuro. However, all local datasets placed in data/raw/ work fully offline. Toggle between sources using the "Local Datasets" / "EEG Dash Datasets" buttons in the Datasets Explorer.
Add your model class in src/models/ following the existing patterns (e.g., cnn.py, transformer.py). Register it in src/models/foundation.py if it's a foundation model. Ensure it accepts standard parameters (n_chans, n_times, n_outputs) and can be trained via the pipeline executor.
Click the Refresh button in the top-right navbar, or directly call GET /api/refresh to clear all server-side caches. The dashboard automatically polls for updates every 30 seconds (configurable via the REFRESH_INTERVAL environment variable).

Key Design Decisions
  • Flask over FastAPI — Simpler setup for template-rendered dashboards with SSE
  • Async via Threading — Pipeline execution uses daemon threads with SSE for real-time progress
  • No Database — Results stored as JSON files on disk for simplicity and reproducibility
  • Numba JIT Disabled — Ensures compatibility with the full braindecode model zoo
  • Fallback Model — SimpleEEGNet provides a reliable baseline when braindecode models fail
  • Dataset Registry — Pluggable loader architecture for adding new datasets
Data Flow
  1. User configures pipeline in the visual builder (browser)
  2. Pipeline graph (JSON) is sent to /api/run-pipeline
  3. Server validates the graph and starts execution in a background thread
  4. Dataset is loaded from local disk or streamed via EEGDash
  5. Preprocessing steps are applied sequentially to the raw EEG data
  6. Processed data is split into train/val/test sets
  7. Selected model is trained with the configured hyperparameters
  8. Evaluation metrics are computed and stored to results/pipeline_runs.json
  9. Progress events are streamed to the browser via SSE throughout the process
